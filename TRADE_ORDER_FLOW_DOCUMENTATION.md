# TRADE AND ORDER RECORDING FLOW DOCUMENTATION

## 🔄 Complete Flow: From Signal to Database Storage

This document explains the complete flow of how trades and orders are recorded in the cache (Redis) and database (PostgreSQL) tables.

## 🎯 PHASE 1: ENTRY CYCLE - ORDER CREATION

### Step 1: Signal Generation
```
[Strategy Service] → Generates trading signal
                  → Sends to Orchestrator Service
                  → Signal contains: pair, exchange, price, amount, strategy, reason
```

### Step 2: Order Placement
```
[Orchestrator Service] → Receives signal
                      → Places BUY limit order on exchange
                      → Gets exchange_order_id from exchange
                      → Creates order data structure
```

### Step 3: Redis Cache Storage (IMMEDIATE)
```redis
# Primary order tracking in Redis
HMSET orders:{order_id}
    trade_id "b4322d60-86d3-4caf-8976-7eb967d3c609"
    symbol "BTC/USDC"
    exchange "binance"
    order_type "limit_buy"
    price "65123.45"
    quantity "1.5"
    status "pending"
    created_at "1693248615.123"
    updated_at "1693248615.123"
    exchange_order_id "binance_order_12345"
    client_order_id "client_order_67890"
    strategy "heikin_ashi"
    entry_reason "Bullish signal detected"

# Indexing for efficient queries
ZADD orders_by_status:pending {timestamp} {order_id}
SADD trade_orders:{trade_id} {order_id}
HSET exchange_orders:{exchange} {order_id} "pending"

# Event stream for reliable processing
XADD order_events *
    action "order_created"
    order_id {order_id}
    trade_id {trade_id}
    status "pending"
    timestamp {timestamp}
    priority "NORMAL"
```

### Step 4: Database Storage (PERSISTENT)
```sql
-- trading.order_mappings table
INSERT INTO trading.order_mappings (
    local_order_id,      -- UUID generated by system
    exchange,            -- 'binance', 'bybit', 'cryptocom'
    exchange_order_id,   -- Order ID from exchange
    client_order_id,     -- Client-generated order ID
    symbol,              -- 'BTC/USDC'
    side,                -- 'buy' or 'sell'
    order_type,          -- 'limit', 'market'
    amount,              -- Order quantity
    price,               -- Order price
    status,              -- 'PENDING', 'FILLED', 'CANCELLED'
    trade_id,            -- Links to trades table
    created_at,
    updated_at
) VALUES (...);
```

## 🎯 PHASE 2: ENTRY CYCLE - ORDER FILL DETECTION

### Step 1: WebSocket Monitoring
```
[WebSocket Service] → Subscribes to order updates
                   → Monitors exchange_order_id
                   → Receives real-time fill events
                   → Validates fill data
```

### Step 2: Fill Event Processing
```redis
# Fill event received via WebSocket
XADD fill_events *
    action "order_filled"
    order_id {order_id}
    exchange "binance"
    symbol "BTC/USDC"
    side "buy"
    executed_quantity "1.5"
    executed_price "65120.00"
    fees "9.77"
    timestamp {timestamp}
    priority "HIGH"
```

### Step 3: Redis Cache Update
```redis
# Update order status in Redis
HMSET orders:{order_id}
    status "FILLED"
    filled_amount "1.5"
    filled_price "65120.00"
    fees "9.77"
    filled_at "1693248700.456"
    updated_at "1693248700.456"

# Move from pending to filled index
ZREM orders_by_status:pending {order_id}
ZADD orders_by_status:filled {order_id} {timestamp}
HSET exchange_orders:{exchange} {order_id} "filled"
```

### Step 4: Database Updates
```sql
-- Update order status
UPDATE trading.order_mappings 
SET status = 'FILLED',
    updated_at = CURRENT_TIMESTAMP
WHERE exchange_order_id = 'binance_order_12345';

-- Create fill record
INSERT INTO trading.fills (
    local_order_id,
    exchange_order_id,
    exchange,
    symbol,
    side,
    qty,
    price,
    fee,
    fee_asset,
    trade_id,
    timestamp
) VALUES (...);
```

### Step 5: Trade Creation
```sql
-- Create OPEN trade record
INSERT INTO trading.trades (
    trade_id,           -- Same UUID from order
    pair,               -- 'BTC/USDC'
    exchange,           -- 'binance'
    entry_price,        -- 65120.00 (actual fill price)
    position_size,      -- 1.5 (actual fill amount)
    status,             -- 'OPEN'
    entry_id,           -- exchange_order_id
    exit_id,            -- NULL (no exit order yet)
    entry_time,         -- Fill timestamp
    exit_time,          -- NULL
    entry_reason,       -- Strategy reason
    strategy,           -- 'heikin_ashi'
    fees,               -- 9.77
    created_at,
    updated_at
) VALUES (...);
```

## 🎯 PHASE 3: EXIT CYCLE - EXIT ORDER CREATION

### Step 1: Exit Signal Detection
```
[Strategy/Orchestrator] → Detects exit signal (profit target, stop loss, trailing stop)
                       → Determines exit price
                       → Creates exit order data
```

### Step 2: Exit Order Placement
```
[Orchestrator Service] → Places SELL limit order on exchange
                      → Gets exit_order_id from exchange
                      → Updates trade with exit_id
```

### Step 3: Redis Cache Update
```redis
# Store exit order in Redis
HMSET orders:{exit_order_id}
    trade_id "b4322d60-86d3-4caf-8976-7eb967d3c609"
    symbol "BTC/USDC"
    exchange "binance"
    order_type "limit_sell"
    price "66000.00"
    quantity "1.5"
    status "pending"
    parent_trade_id "b4322d60-86d3-4caf-8976-7eb967d3c609"
    created_at "1693248800.789"

# Update trade state
HMSET trade:{trade_id}
    exit_order_id {exit_order_id}
    trailing_active "true"
    last_trail_update {timestamp}

# Add to pending orders
ZADD orders_by_status:pending {exit_order_id} {timestamp}
```

### Step 4: Database Updates
```sql
-- Create exit order record
INSERT INTO trading.order_mappings (
    local_order_id,
    exchange,
    exchange_order_id,  -- exit_order_id
    client_order_id,
    symbol,
    side,               -- 'sell'
    order_type,
    amount,
    price,
    status,             -- 'PENDING'
    trade_id,           -- Links to same trade
    created_at,
    updated_at
) VALUES (...);

-- Update trade with exit_id
UPDATE trading.trades 
SET exit_id = 'exit_order_12345',
    updated_at = CURRENT_TIMESTAMP
WHERE trade_id = 'b4322d60-86d3-4caf-8976-7eb967d3c609';
```

## 🎯 PHASE 4: EXIT CYCLE - EXIT ORDER FILL DETECTION

### Step 1: WebSocket Monitoring
```
[WebSocket Service] → Monitors exit_order_id
                   → Receives exit fill event
                   → Validates exit fill data
```

### Step 2: Exit Fill Processing
```redis
# Exit fill event
XADD fill_events *
    action "exit_order_filled"
    order_id {exit_order_id}
    trade_id {trade_id}
    executed_quantity "1.5"
    executed_price "65950.00"
    fees "9.89"
    timestamp {timestamp}
    priority "HIGH"
```

### Step 3: Redis Cache Updates
```redis
# Update exit order status
HMSET orders:{exit_order_id}
    status "FILLED"
    filled_amount "1.5"
    filled_price "65950.00"
    fees "9.89"
    filled_at {timestamp}

# Update trade state
HMSET trade:{trade_id}
    status "CLOSED"
    exit_price "65950.00"
    exit_time {timestamp}
    realized_pnl "1245.00"  -- Calculated PnL
    last_update {timestamp}

# Move to completed indexes
ZREM orders_by_status:pending {exit_order_id}
ZADD orders_by_status:filled {exit_order_id} {timestamp}
ZREM active_trades {trade_id}
ZADD completed_trades {trade_id} {timestamp}
```

### Step 4: Database Final Updates
```sql
-- Update exit order status

    price,
    fee,

    timestamp
```

## 🗄️ DATABASE TABLES STRUCTURE

### 1. trading.trades
```sql
CREATE TABLE trading.trades (
    id SERIAL PRIMARY KEY,
    trade_id UUID UNIQUE NOT NULL,
    pair VARCHAR(50) NOT NULL,           -- 'BTC/USDC'
    exchange VARCHAR(50) NOT NULL,       -- 'binance'
    entry_price NUMERIC(20,8),           -- Actual entry price
    exit_price NUMERIC(20,8),            -- Actual exit price
    status VARCHAR(20) NOT NULL,         -- 'OPEN', 'CLOSED', 'FAILED'
    entry_id VARCHAR(255),               -- Entry order ID
    exit_id VARCHAR(255),                -- Exit order ID
    entry_time TIMESTAMP WITH TIME ZONE,
    exit_time TIMESTAMP WITH TIME ZONE,
    position_size NUMERIC(20,8),         -- Actual position size
    unrealized_pnl NUMERIC(20,8),        -- Current PnL (for OPEN trades)
    realized_pnl NUMERIC(20,8),          -- Final PnL (for CLOSED trades)
    fees NUMERIC(20,8),                  -- Total fees
    strategy VARCHAR(50),                -- Strategy name
    entry_reason TEXT,                   -- Why trade was opened
    exit_reason TEXT,                    -- Why trade was closed
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
```

### 2. trading.order_mappings
```sql
CREATE TABLE trading.order_mappings (
    id SERIAL PRIMARY KEY,
    local_order_id UUID NOT NULL,        -- System-generated UUID
    exchange VARCHAR(50) NOT NULL,       -- 'binance', 'bybit', 'cryptocom'
    exchange_order_id VARCHAR(255),      -- Order ID from exchange
    client_order_id VARCHAR(255) NOT NULL, -- Client-generated ID
    symbol VARCHAR(50) NOT NULL,         -- 'BTC/USDC'
    side VARCHAR(10) NOT NULL,           -- 'buy' or 'sell'
    order_type VARCHAR(20) NOT NULL,     -- 'limit', 'market'
    amount NUMERIC(20,8) NOT NULL,       -- Order quantity
    price NUMERIC(20,8),                 -- Order price
    status VARCHAR(20) NOT NULL DEFAULT 'PENDING', -- 'PENDING', 'FILLED', 'CANCELLED'
    trade_id UUID REFERENCES trading.trades(trade_id),
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    acknowledged_at TIMESTAMP WITH TIME ZONE
);
```

### 3. trading.fills
```sql
CREATE TABLE trading.fills (
    id SERIAL PRIMARY KEY,
    local_order_id UUID NOT NULL,
    exchange_order_id VARCHAR(255),
    exchange VARCHAR(50) NOT NULL,
    symbol VARCHAR(50) NOT NULL,
    side VARCHAR(10) NOT NULL,
    qty NUMERIC(20,8) NOT NULL,          -- Actual filled quantity
    price NUMERIC(20,8) NOT NULL,        -- Actual fill price
    fee NUMERIC(20,8) DEFAULT 0,         -- Fee paid
    fee_asset VARCHAR(10) DEFAULT 'USDC',
    trade_id UUID,
    timestamp TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
```

## 🔄 REDIS CACHE STRUCTURE

### 1. Order State Management
```redis
# Primary order storage
orders:{order_id} → Hash with all order data
orders_by_status:{status} → Sorted set of order IDs by timestamp
exchange_orders:{exchange} → Hash of order_id → status
trade_orders:{trade_id} → Set of order IDs for a trade
```

### 2. Trade State Management
```redis
# Trade state storage
trade:{trade_id} → Hash with trade data
active_trades → Sorted set of active trade IDs
completed_trades → Sorted set of completed trade IDs
trades_by_symbol:{exchange}:{symbol} → Set of trade IDs
```

### 3. Event Streams
```redis
# Reliable event processing
order_events → Stream of order lifecycle events
fill_events → Stream of fill events
price_updates → Stream of price updates
```

## 🚨 CRITICAL FLOW POINTS

### 1. Order Creation Flow
```
Strategy Signal → Orchestrator → Exchange Service → Redis Cache → Database
     ↓                ↓              ↓                ↓           ↓
  Signal Data    Order Request   Exchange Order   Cache Store  DB Record
```

### 2. Fill Detection Flow
```
WebSocket Event → Redis Manager → Database Service → Trade Creation
      ↓               ↓               ↓                ↓
  Fill Data      Cache Update    Order Update    Trade Record
```

### 3. Exit Cycle Flow
```
Exit Signal → Exit Order → WebSocket Monitor → Fill Detection → Trade Closure
     ↓            ↓              ↓                ↓              ↓
  Exit Logic   Order Place   Monitor Order    Detect Fill   Close Trade
```

## 🔧 CONFIGURATION IMPACT

The configuration settings in `config.yaml` affect this flow:

```yaml
order_management:
  order_sync_interval_seconds: 15  # How often to sync with exchange
  order_status_cache_seconds: 30   # Redis cache TTL
  stale_order_cleanup_hours: 12    # When to clean up old orders
  sync_retry_attempts: 3           # Retry failed syncs
  websocket_fallback_enabled: true # Use WebSocket for fill detection
  polling_fallback_interval: 60    # Fallback polling interval
```

## 📊 MONITORING AND ALERTS

The system monitors:
- **Pending orders** that should be filled
- **OPEN trades** with exit_id (should be 0)
- **Order sync failures** between cache and database
- **WebSocket connectivity** issues
- **Database service** health

Alerts are triggered when:
- More than 10 pending orders for >1 hour
- Any OPEN trades with exit_id
- Order sync service failures
- Service connectivity issues

## 🎯 SUMMARY

The complete flow ensures:
1. **Real-time tracking** via Redis cache
2. **Persistent storage** in PostgreSQL database
3. **Reliable fill detection** via WebSocket monitoring
4. **Automatic trade lifecycle** management
5. **Comprehensive monitoring** and alerting

This architecture provides both speed (Redis cache) and reliability (PostgreSQL database) while maintaining data consistency across all components.
